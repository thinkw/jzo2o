### DeepSeek 思考模式兼容
- 问题：`deepseek-v4-flash` 默认开启 thinking 模式，`reasoning_content` 在工具调用往返中丢失，导致 400 错误
- 根因：`langchain-openai 1.2.1` 的 `_convert_message_to_dict` 不序列化 `additional_kwargs.reasoning_content`
- 解决：
  - 入站方向：用 `langchain-deepseek` 的 `ChatDeepSeek`（官方集成，原生提取 `reasoning_content`）
  - 出站方向：monkey-patch `_convert_message_to_dict` 将 `reasoning_content` 写回请求体
- 参考：[DeepSeek Thinking Mode 文档](https://api-docs.deepseek.com/guides/thinking_mode) — 工具调用后必须传回 `reasoning_content`

## 二、评价 AI 总结功能

### 需求
- 定时任务（每周）+ 管理员手动触发
- 增量模式：AI 先读上次总结，从上次时间点往后读新评价，合并生成新总结
- 前端企业端评价管理页新增"AI 总结"按钮

### 数据模型
- 新表 `evaluation_summary`（建在 jzo2o-ai 库）
- 字段：target_type_id, target_id, summary_content, last_evaluation_id, last_evaluation_time
- 唯一索引：`(target_type_id, target_id)`

### 架构分层
```
前端"AI 总结"按钮
  → GET /customer/agency/evaluation/summarize (AgencyEvaluationController)
    → AiApi.getEvaluationSummary() [Feign → jzo2o-ai]
      → EvaluationSummaryService.getSummary() → 查库
      → 无已有总结 → AiApi.summarizeEvaluation() [Feign → jzo2o-ai]
        → EvaluationSummaryService.summarize()
          → EvaluationApi.queryByTargetIdAndTime() [Feign → jzo2o-customer]
          → AiEngineWebSocketClient.connectAndCollect() [WebSocket → Python Agent]
          → 保存/更新 evaluation_summary
```

## 三、踩坑记录

### 1. Java 11 兼容性
- `instanceof` 模式匹配（`val instanceof Integer i`）是 Java 16+ 特性，改为传统写法
- MyBatis-Plus 无 `insertOrUpdate()` 方法，改为 `insert` / `updateById` 分支

### 2. Axios POST + params 陷阱
- 请求 wrapper 对 POST 无 `data` 时会把 `params` 移入请求体
- `@RequestParam` 找不到参数报 `MissingServletRequestParameterException`
- 修复：前端改用 GET，后端合并 GET/POST 逻辑

### 3. 响应双重序列化
- Spring 返回 `String` 时，`PackResultFilter` 会双重序列化
- 前端 `res.data` 拿到的是 JSON 字符串而非对象
- 修复：前端 `JSON.parse` 解析内层 JSON

### 4. 前端超时
- AI 总结生成需 ~20 秒，前端默认 10 秒超时
- 修复：该 API 单独设置 180 秒超时

### 5. JSON 时间戳类型
- Hutool 序列化 `LocalDateTime` 为 Long 时间戳，非字符串
- `(String) lastEval.get("createTime")` 抛 `ClassCastException`
- 修复：按 Number/String 分支处理，区分秒/毫秒级时间戳
